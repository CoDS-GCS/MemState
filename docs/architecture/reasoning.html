Reasoner behavior

The reasoner is a rule-based maintenance engine invoked on lifecycle events. It uses the same executor/store objects as request handlers, but it runs after the client response for ingest and query endpoints.

Design intent

  • Keep core API semantics stable while allowing memory-maintenance policy to evolve independently.
  • Run lightweight bounded heuristics instead of opaque models for cleanup tasks.
  • Avoid adding request latency by executing maintenance in background callbacks.

Event to action matrix

Event Forget action Revise action
ingest_complete Runs if topic count >= topic_count_soft_limit. Runs duplicate-title revise when total topic count is positive.
query_complete Runs if topic count >= topic_count_soft_limit. Not executed for this event.
cron Runs if topic count >= soft limit. Runs duplicate-title revise when topic count is positive.
memory_pressure Always runs forget (independent of count threshold). Not executed for this event.

Decision flow diagram

Reasoner returns a structured ReasonerResult containing lists of revise and forget actions.

Forget and revise semantics

Action Implementation behavior Effect
Forget low salience Scans non-archived topics and archives entries with salience below forget_salience_threshold. Marks archived=true and halves salience on affected topics.
Revise duplicates Uses title equality heuristic (find_duplicate_titles) and merges pairs. Moves edges/fields toward keep topic and deletes drop topic.

Operational implications

  • Large ingest bursts can trigger cleanup soon after writes, because ingest_complete runs every request.
  • Duplicate detection is title-based; unrelated topics with identical titles may be merged unless upstream naming is careful.
  • If maintenance policy should not run, reasoner invocation must be disabled in route handlers or guarded by custom conditions.
