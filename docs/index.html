MemState

Agent memory infrastructure

Durable, evolving memory for AI agents.

MemState is a long-term memory service that keeps agent knowledge coherent as it grows. It organizes what the agent learns, tracks how facts change, prunes what stops mattering, and returns the right context on every retrieval.

What MemState is

MemState is a single FastAPI service backed by an embedded Kuzu graph. It exposes three access surfaces on one shared store: a product API (/v1), a low-level graph API (/api/ui), and a natural-language tool loop (/api/llm/chat). It is designed for one operational idea: memory as a managed, evolving store rather than a passive collection of records.

Why it exists

Agents need memory that does more than store records. As the agent talks, learns, and works over time, what it remembers has to evolve: new input has to be placed correctly, old beliefs have to be updated when evidence changes, unused knowledge has to decay, and every retrieval has to surface the current truth without throwing away how it got there.

Classical stores were not built for this. A plain database overwrites. A vector database ranks by similarity but has no notion of freshness or supersession. A document store can't safely reason about cross-record consequences. MemState treats these as first-class concerns of the memory system rather than problems to solve in application code.

The four memory operations

All behavior in MemState is expressed through four operations. Two are triggered by clients; two run in the background to keep the store healthy.

Operation Triggered by What it does Access
Ingest Client / agent tool Integrate new input into the store. Decide placement, append a new field value, or create a new topic. Write
Retrieve Client / agent tool Answer a question with the right topic, its neighbors, and the right version of each field. Read (with reinforcement)
Revise Background (after ingest) Reconcile contradictions, consolidate duplicates, and mark superseded values. Internal
Forget Background (pressure / schedule) Attenuate, archive, or compress content that is no longer earning its place. Internal
Retrieval is not purely read-only. Every successful retrieval reinforces the topics it touches by increasing their salience. This is how the system learns what stays active and what can be forgotten.

How memory is organized

MemState organizes memory into topics. Every topic is a self-contained unit of knowledge: a title, a summary, a set of typed fields, an embedding for semantic search, and typed links to other topics. You can read one topic and get the full context of one thing the agent knows, without following ten hops across a generic graph.

One topic node carries all the context about one thing. Links model relationships; history preserves how facts evolved.

How a request moves through the system

  1. Client or agent calls the API. Ingest for writes, retrieve for reads. No direct database access — every mutation goes through one semantics layer.
  2. The Executor applies the operation. It resolves placement (new topic, extend, or append), runs the lowest-level primitives against the store, and enforces policy limits.
  3. The Reasoner runs in the background. When ingest finishes or pressure builds, the reasoner triggers revise and forget to keep the store coherent and lean.
  4. The response returns. Every touched topic has its salience updated, so the next retrieval already reflects what was just read or written.

Clients never touch the store directly. All writes and reads flow through the Executor; background maintenance runs through the Reasoner.

Correctness promises

MemState keeps two guarantees that make it safe to depend on as the system of record for agent knowledge.

Promise 1 — Current truth by default Every retrieval returns the most recent, non-archived value for each field. Older values are never returned as current — they surface only when a caller explicitly asks for history.
Promise 2 — Transitions respect policy Every change to the store goes through the ingest, revise, or forget path. Limits on history length, salience thresholds, and archival rules are enforced uniformly — there is no back door that bypasses them.

Quick start

Run MemState locally:

python -m pip install -e .
uvicorn memstate.api.main:app --reload

Write a topic:

curl -X POST http://localhost:8000/v1/ingest \
  -H "content-type: application/json" \
  -d '{
    "placement": "new_topic",
    "title": "Launch plan",
    "summary": "Q3 public release milestones",
    "fields": [
      {"name": "owner", "value": "Aya", "provenance": "ui"},
      {"name": "target_date", "value": "2026-07-01", "field_type": "date"}
    ]
  }'

Read it back:

curl -X POST http://localhost:8000/v1/query \
  -H "content-type: application/json" \
  -d '{ "q": "when is launch?", "top_k": 5 }'
