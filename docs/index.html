MemState official system documentation

MemState is a single-process memory service built on FastAPI and embedded Kuzu. It stores structured, versioned knowledge as a topic graph and exposes it through three access surfaces on one shared store: /v1 for product memory semantics, /api/ui for low-level graph operations, and /api/llm/chat for tool-based natural language access.

This site is the official engineering reference for maintainers and integrators. Every behavior described here is mapped to the current codebase under src/memstate.

Data model: the Topic graph

MemState stores knowledge as a topic graph: each Topic is a single versioned unit with typed fields (each with per-field history), one embedding vector, salience-driven retention, and explicit RELATED edges for structure.

Home page stays high-level; the full rationale and the Topic anatomy diagram live in Data model / Topic model.

Low-level operations: GraphStore primitives

Internally, all behavior decomposes into a small set of primitive operations implemented in memstate.store.graph_store. These are the only code paths that issue Cypher and touch the embedded database directly.

Start with the primitive reference if you're implementing new behavior, debugging Cypher, or tracing side effects.

API / Endpoint index is the easiest way to find the UI and graph routes for calling primitives from tools.

Primitive operations reference

Topic create_topic INSERT a new Topic node with scalar properties and embedding
get_topic Fetch one topic row including fields_json, topic_history_json, embedding
update_topic_meta SET scalar properties (title, summary, salience, archived, topic_kind); does not touch fields
update_topic_embedding SET embedding vector only; called after title/summary changes

Field create_field Initialize a named field entry in fields_json with type, optional ref, optional first value
append_field_history Prepend a new FieldHistoryEntry to the field's history list; enforce max_history cap
get_field_with_history Return the full field including all history entries
delete_field Remove field key from fields_json
set_field_ref Point a field's ref_topic_id at another topic (enables expansion)
Structure promote_fields_to_nested_topic Move selected fields to a new child topic; add RELATED edge + optional parent ref field
undo_promote_nested_topic Merge child fields back to parent; delete child topic and RELATED edge
nest_fields_in_topic Collapse selected fields into one json bundle field on the same topic (no new topic)
unnest_fields_in_topic Expand a bundle field back to top-level fields

remove_relationship DELETE a specific RELATED edge by (from, to, kind)

High-level operations: ingest, query, revise, forget

Product clients usually interact with MemState through semantic operations that compose the primitives into a coherent workflow. Each operation is documented in the Operations section:

  • Ingest: write or extend knowledge; compute embeddings; set salience; create edges.
  • Query: multi-stage retrieval (semantic + structural + temporal) with salience bump-on-read.
  • Revise: internal maintenance (duplicate merging) triggered after ingest.
  • Forget: internal archiving pass driven by salience and memory pressure.

System design

The three user-facing surfaces converge on one shared GraphStore and one embedded Kuzu file.

Source map

  • memstate.api.main: app creation, route registration, auth dependency, health endpoints.
  • memstate.api.deps: singleton wiring for Executor, Reasoner, and shared GraphStore.
  • memstate.core.executor: all /v1 ingest and query semantics, embedding, salience bumping.
  • memstate.store.graph_store and store.kuzu_adapter: all Cypher primitives, schema bootstrap, Kuzu access layer.
  • memstate.reasoner.engine: event-based policy loop (forget + duplicate revise).
  • memstate.llm.chat_api and llm.tool_runner: routed chat and tool execution.
  • memstate.config: all tunable thresholds — forget_salience_threshold, topic_count_soft_limit, max_field_history, query_salience_bump, embedding_dimension.
