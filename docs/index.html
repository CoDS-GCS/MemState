MemState

MemState is the reference implementation of Governed Evolving Memory (GEM), a data-management model for long-term AI agent memory. It replaces record-level CRUD semantics with four state-level operators ingestion, revision, forgetting, and retrieval acting over a governed evolving state. The system is packaged as a single FastAPI process backed by an embedded Kuzu property graph, and exposes three access surfaces on one shared store: /v1 for product semantics, /api/ui for low-level graph operations, and /api/llm/chat for tool-based natural language access.

This site is the engineering reference for maintainers and integrators. Every behavior described here is mapped to the current codebase under src/memstate; every abstraction is traceable to the MemState paper, Is Agent Memory a Database?, which this documentation treats as the source of truth for definitions and operators.

Governed evolving memory

Classical databases localize correctness at the record or transaction level and expose primitive storage operations (insert, update, delete, select). This is insufficient for agent memory, which must integrate new input into existing knowledge, reconcile internal contradictions, attenuate low-relevance content under policy control, and reinforce relevance through access. MemState models memory as a governed evolving state whose content, structure, and policies change continuously over time.

Memory is modeled as Mt = (Dt, St, Pt) and evolves through four state-level operators under explicit policy control.

Memory state (Definition 1)

At time t, the agent's memory state is the tuple Mt = (Dt, St, Pt), where:

  • Dt is the stored informational content organized as semantic units (topics, in MemState's instantiation).
  • St is the structural organization over that content (typed relations, temporal order, associations).
  • Pt is the set of evolution policies of the form ⟨event, condition, action⟩ (Definition 3) that govern access, ingestion, revision, and forgetting.

Memory evolves according to the state transition Mt+1 = U(Mt, It, Rt), where It denotes new external inputs at time t and Rt denotes internal operations that may alter memory state (Definition 2). See Data model / State abstraction for the MemState instantiation.

The four state-level operators

Operator Formal transition Role in memory evolution Access
Ingestion (Def. 4) Mt+1 = U(Mt, It, ∅) Integrate input It into existing semantic units rather than appending independent records. Client-facing (synchronous)
Revision (Def. 5) Mt+1 = U(Mt, ∅, rev(Δ)) Reorganize memory from internal evidence Δ: consolidate, reconcile, repair. Internal (policy-triggered)
Forgetting (Def. 6) Mt+1 = U(Mt, ∅, F) Regulate influence of stored content by relevance; non-destructive graduated attenuation. Internal (policy-triggered)
Retrieval (Def. 7) R(Mt, q) → o, may induce U(Mt, ∅, Rq) Map query to output and reinforce relevance of accessed content (read-write, not read-only). Client-facing (synchronous)

MemState exposes these operators through /v1/ingest, /v1/query, and internal reasoner maintenance. Each operator is documented on its own page under Operations.

Memory correctness (Definition 8)

Correctness in persistent agent memory cannot be defined at the record level because contradictory or outdated entries may legitimately coexist within Dt. Let ui ∈ Dt be a semantic unit with a time-ordered value history Vi = ⟨(v1, t1, π1), …, (vk, tk, πk)⟩, where each entry records a value v, a timestamp t, and provenance π. A memory system is correct if:

  1. Query soundness. For every query q and state Mt, the response R(Mt, q) → o reflects the most recent non-archived value vk as current; prior values appear only when q explicitly requests historical or provenance context.
  2. Transition soundness. Every state transition Mt → Mt+1 produced by U respects policy constraints in Pt, and no revision produces a state in which a superseded value is returned as current on a subsequent query.

MemState: realizing GEM on Kuzu

Among classical data models, the property graph is the closest substrate for GEM because it supports typed nodes, typed edges, and indexed properties. MemState builds on Kuzu and extends it along three axes that a bare property graph does not provide:

  • Self-contained semantic units. Each topic node materializes one coherent unit (title, summary, fields, histories, embedding) so one read returns full context without multi-hop traversal.
  • Field-level value history. Every field keeps a time-ordered history Hi,j = ⟨(v, t, π)⟩ with value, timestamp, and provenance, enabling point-in-time queries and preserving evidence required for revision.
  • Co-located semantic ranking. Embeddings are stored on the topic node itself, so similarity search and graph traversal operate over one consistent record with no cross-store drift.

Structure St links topic nodes through typed RELATED edges: extension edges connect topics in the same broader context, while association edges connect related but independent topics. Policies Pt are declarative rules that govern when the graph evolves through transitions such as splitting oversized topics, linking related topics, or compressing and archiving low-salience content.

Primitive operations reference

Every state-level operator decomposes into a fixed set of primitives: typed single-step storage actions organized at three levels (Section 5.2 of the paper). Primitives are the only code paths that issue Cypher and touch the embedded database directly, and they are reused across operators to make evolution explicit, composable, and verifiable.

LevelPrimitiveWhat it does
Field level create_field Initialize a named field entry in fields_json with type and optional initial value.
append_field_history Prepend a FieldHistoryEntry (v, t, π) to Hi,j; enforce max_field_history.
get_field_with_history Return the full field including all history entries.
bump_field_salience_on_query Increment field saliences (capped) and re-sync topic salience; backs retrieval reinforcement.
set_field_ref Point a field's ref_topic_id at another topic, enabling expansion.
Topic level create_topic Insert a new Topic node with scalar properties and embedding.
get_topic Fetch one topic row including fields_json, topic_history_json, embedding.
update_topic_meta / update_topic_embedding Set scalar properties (title, summary, salience, archived, topic_kind) or embedding vector.
merge_topics Transfer fields and edges from drop topic into keep topic, then detach-delete drop (revision primitive).
promote_fields_to_nested_topic / undo_promote_nested_topic Split an overloaded topic into parent/child, or fold a child back; reversible.
Graph level add_related_edge Create or merge a RELATED edge with a named kind string (extension or association).
remove_relationship Delete a specific RELATED edge by (from, to, kind).
list_relationships Return neighbor topic IDs and edge kinds, filtered by direction.

See Operations / Low-level UI API for the HTTP surface that exposes these primitives directly to editors and tooling.

Source map

  • memstate.api.main: app creation, route registration, auth dependency, health endpoints.
  • memstate.api.deps: singleton wiring for Executor, Reasoner, and shared GraphStore.
  • memstate.core.executor: ingestion placement semantics and retrieval composition over primitives; embedding and salience reinforcement.
  • memstate.store.graph_store and store.kuzu_adapter: all Cypher primitives, schema bootstrap, Kuzu access layer.
  • memstate.reasoner.engine: policy-triggered revision and forgetting loop.
  • memstate.llm.chat_api and llm.tool_runner: routed chat and tool execution.
  • memstate.config: all tunable thresholds — forget_salience_threshold, topic_count_soft_limit, max_field_history, query_salience_bump, embedding_dimension.
