Operation: revision (internal, policy-triggered)

Revision is an internal maintenance operation. In code, its objective is to reduce obvious duplication by merging topics that share the same non-empty title, using store-level merge rules.

Access and trigger

  • Not client-accessible: no public HTTP route directly calls revision.
  • Triggered by reasoner: Reasoner.run() invokes revision on ingest_complete and cron when topic count is positive.
  • Path: reasoner.run(event) -> executor.run_revise_duplicates().
Current behavior: revision runs only when reasoner is invoked. In stock API code that happens after /v1/ingest and /v1/query responses. UI API, LLM tool writes, and MCP tool writes do not auto-trigger revision.

Objective in code

Objective: collapse duplicate-title topic pairs to keep memory graph less fragmented, while preserving reachable edges/fields on the surviving topic.

Visual flow (current)

Revision currently uses duplicate title detection followed by merge transfer operations.

Revision algorithm (current implementation)

  1. Find duplicate-title pairs via GraphStore.find_duplicate_titles().
  2. For each pair (title, id_a, id_b), call merge_topics(keep_id=id_a, drop_id=id_b).
  3. Record action string: merged:{drop_id}->:{keep_id}:{title}.

Duplicate candidate criteria

  • Only pairs where a.id < b.id and a.title == b.title.
  • Empty-title topics are excluded.
  • Result set is limited to 50 pairs per run.

Merge behavior details

Merge step What code does
Outgoing RELATED edges Rewrites edges from drop topic to keep topic using MERGE, then deletes old edges.
Incoming RELATED edges Rewrites incoming edges to point to keep topic, then deletes old edges.
fields_json merge Copies only missing field names from drop into keep; existing-name conflicts are left unchanged.
Legacy field chains Moves legacy HAS_FIELD heads when field name absent on keep topic.
Drop topic removal DETACH DELETE drop topic after transfer steps.

Execution sequence

reasoner event (ingest_complete or cron)
  -> Reasoner.run(event)
  -> Executor.run_revise_duplicates()
  -> GraphStore.find_duplicate_titles()
  -> for each duplicate pair: GraphStore.merge_topics(keep, drop)
  -> return revise_actions[] in ReasonerResult

Current limitations and implications

  • Revision heuristic is title-only. It does not compare summaries, fields, embeddings, or entity identity.
  • Field merge is "missing-name only"; same-name field histories are not merged today.
  • No client-facing dry-run endpoint exists for revision decisions.
