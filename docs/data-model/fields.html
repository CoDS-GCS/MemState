Fields and value history

Fields are the fine-grained state carried by each topic. A topic ui exposes a name-keyed map of fields fi,j in fields_json; each field carries type metadata, optional topic reference, per-field salience, and an ordered value history (newest first).

Value history Hi,j

For each field fi,j of topic ui, MemState maintains a time-ordered value history Hi,j = ⟨(v1, t1, π1), …, (vk, tk, πk)⟩, where each entry records a value v, a timestamp t, and the provenance π of the source interaction that produced it (for example, a user message or a revision event). This is the primary evidence structure that supports query soundness (Definition 8): the system distinguishes the current value vk from prior values v1, …, vk−1 and returns historical values only when the query explicitly asks for them.

Why histories and not overwrites. Overwriting a field value destroys Hi,j and irreversibly truncates Vi, violating query soundness for any historical or provenance query. Ingestion therefore appends rather than overwrites; revision marks superseded entries without deleting them.

Field record schema

{
  "field_name": {                            // f_{i,j}
    "field_type": "string|int|float|bool|date|datetime|list|json",
    "ref_topic_id": "optional-topic-id",     // pointer into S_t
    "salience": 1.0,                         // reinforced by retrieval
    "history": [                             // H_{i,j}, newest-first
      {
        "id": "uuid",
        "valid_from": "2026-03-24T12:00:00+00:00",   // t
        "value": "...",                               // v
        "why_changed": "optional",
        "impact_expected": "optional",
        "provenance": "api|ui|llm|mcp|..."            // π
      }
    ]
  }
}

Temporal semantics

  • History order is newest-first; history[0] is the current value vk.
  • Append operations always create a new revision id (UUID) when absent.
  • History length is capped by Policies.max_field_history (default 500); old revisions are truncated from the tail when the cap is exceeded.
  • Provenance π identifies the source channel (api, ui, llm, mcp, internal revision event, …).

Field salience

  • Each field carries a salience float in [0, 10]. Invalid values are clamped in validators.
  • Retrieval reinforces salience on read; forgetting consults it on policy evaluation.
  • Topic salience is synchronized to the mean of field saliences after field mutations (sync_topic_salience_from_fields).

Revision lifecycle

Incoming write (UI API, /v1 ingest, or LLM tool)
  -> build FieldHistoryEntry (id, valid_from=t, value=v, provenance=π)
  -> append_field_history(topic_id, field_name, entry)
  -> insert at history index 0
  -> trim to max_field_history
  -> persist fields_json
  -> sync topic salience from field salience average
