Fields and revision history

Fields are stored as a name-keyed map inside fields_json. Each field includes type metadata, optional topic reference, per-field salience, and an ordered revision history (newest first).

Field record schema

{
  "field_name": {
    "field_type": "string|int|float|bool|date|datetime|list|json",
    "ref_topic_id": "optional-topic-id",
    "salience": 1.0,
    "history": [
      {
        "id": "uuid",
        "valid_from": "2026-03-24T12:00:00+00:00",
        "value": "...",
        "why_changed": "optional",
        "impact_expected": "optional",
        "provenance": "api|ui|llm|mcp|..."
      }
    ]
  }
}

Temporal semantics

  • History order is newest first; history[0] is current value.
  • Append operations always create a new revision id when absent.
  • History length is capped by Policies.max_field_history (default 500).
  • Old revisions are truncated from the tail when cap is exceeded.

Field salience behavior

  • Each field stores a salience float in [0, 10]. Invalid values are clamped in validators.
  • Query-route LLM read tools can bump field salience (configurable via settings).
  • Topic salience may be synchronized to the average of field saliences after field mutations.

Revision lifecycle diagram

Incoming write (UI API, /v1 ingest, or LLM tool)
  -> build FieldHistoryEntry (id, valid_from, value, provenance)
  -> append_field_history(topic_id, field_name, entry)
  -> insert at history index 0
  -> trim to max_field_history
  -> persist fields_json
  -> sync topic salience from field salience average
