Limits, thresholds, and configuration

MemState exposes governance through two layers: Policies (Pt defaults consumed by executor and reasoner) and Settings (runtime options driven by environment variables). The forgetting policy in particular is expressed as a ladder of thresholds — see Forgetting ladder.

Forgetting ladder (paper definition vs. current prototype)

The paper defines forgetting as a three-stage attenuation ladder parameterized by salience thresholds θsummary > θremove > θarchive. The current MemState prototype implements the archival stage and treats the remaining stages as future work.

Stage Paper threshold Effect on state Prototype support
Summarize θsummary Compress older history into summaries while keeping the topic active. Future
Detach / remove from active working set θremove Detach less-salient fields or connectors from active retrieval; retain for explicit lookup. Future
Archive θarchive Mark topic as archived so it is excluded from default active scans and retrieval candidates. Implemented via forget_salience_threshold and archived flag.

Forgetting is non-destructive: archived topics and their field histories remain in the store and are excluded only from default active scans. Explicit queries targeting archived topics can still reach them, which is what the paper calls non-destructive attenuation by relevance.

Retrieval stages

POST /v1/query accepts any subset of semantic, structural, and temporal; the default is all three. These stages implement the three retrieval modes described in the paper.

Stage Retrieval mode What the executor does Output effect
semantic Topic retrieval Embeds query, runs vector KNN (or cosine fallback) over topic embeddings. Ranked candidate list with similarity scores.
structural Structural retrieval Expands candidates via RELATED neighbors and selected fields. Populates neighbors[] and field entries.
temporal Temporal retrieval Enables field-history expansion when combined with explain=true. Returns full Hi,j instead of current-value snapshot.

Stage interaction

query request
  -> build candidate topic ids
       semantic on  -> vector/cosine ranking
       semantic off -> topic_ids allowlist or embedding scan order
  -> for each candidate topic
       structural on -> neighbors + selected fields
       temporal on + explain=true -> full field history (H_{i,j})
       otherwise -> current field snapshot only
  -> salience bump (reinforcement) and topic history append
  -> response summary_text + candidates[]

Practical limits and caveats

  • top_k bounds the number of candidate topics returned by retrieval.
  • version_field ingest placement requires exactly one field and a valid topic_id.
  • Large nested json values are allowed, but payload size affects retrieval latency.
  • Archived topics are retained; forgetting is non-destructive (paper, Definition 6).
