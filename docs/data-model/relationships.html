Relationships and field refs

Structure St links topic nodes so that updates propagate appropriately and so that retrieval can expand context beyond a single semantic unit. MemState represents St through two complementary mechanisms: typed RELATED edges between topics and field-level pointers via ref_topic_id.

Canonical edge kinds

MemState classifies RELATED edges into two canonical kinds that mirror the structural semantics of the paper. Edge kind is a named string, but inter-topic links should map to one of these two intents.

Edge intent Meaning Typical kind strings When to use
Extension Connects topics in the same broader context; the target extends or specializes the source (for example, a sub-project, a deliverable, a session within a workstream). has_detail, has_subtopic, part_of, extends When information about the source topic naturally decomposes into a child topic that retains the parent's context.
Association Connects related but independent topics whose contexts are separate (for example, a person and a project they touched, or two concepts that interact). associated_with, mentions, collaborates_with, references When two topics share meaningful relationships without one subsuming the other.

RELATED edges versus field refs

Mechanism Stored as Best for
RELATED edge Kuzu relationship (Topic)-[:RELATED {kind}]->(Topic) Explicit graph traversal, neighborhood queries, structural retrieval.
ref_topic_id Field attribute inside fields_json Document-style references embedded in attribute payload; preserved across history entries.

RELATED semantics

  • Directed. An edge is written from from_id to to_id.
  • Idempotent key. MERGE yields at most one edge for a given (from, to, kind).
  • Multi-kind. The same pair may carry several relationship kinds.
  • Retrieval impact. Structural retrieval and UI graph builders read these edges directly to expand context.

ref_topic_id semantics

  • Pointer lives on a field record, not as a native edge in Kuzu tables.
  • Useful when a field value naturally points to another unit (for example, manager.ref_topic_id pointing to a person topic).
  • UI graph snapshots synthesize field_ref edges so refs are visible in visualization.
  • The REF_UNCHANGED sentinel allows appending field history without overwriting the existing ref.

Guidance

Use case Prefer Reason
Parent/child context extension RELATED (extension) Communicates structural containment for retrieval expansion.
Peer-to-peer association between units RELATED (association) Explicit edge semantics with multi-kind support.
Attribute-level pointer inside one field ref_topic_id Keeps reference close to field meaning and revision history.
Typed reference and graph traversability Both ref_topic_id for schema meaning; RELATED for graph operations.
