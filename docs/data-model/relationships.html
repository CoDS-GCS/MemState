Relationships and field refs

MemState supports two complementary linking mechanisms: graph-native RELATED edges and field-level pointers via ref_topic_id.

Mechanism comparison

Mechanism Stored as Best for
RELATED Kuzu relationship (Topic)-[:RELATED {kind}]->(Topic) Explicit graph traversal and neighborhood queries.
ref_topic_id Field attribute inside fields_json Document-style references embedded in attribute payload.

RELATED semantics

  • Directed by storage: edge is written from from_id to to_id.
  • Idempotent key: MERGE yields at most one edge for a given (from, to, kind).
  • Multi-kind support: same pair can have several relationship kinds.
  • Neighborhood queries: structural query stage and UI graph builder read these edges directly.

ref_topic_id semantics

  • Pointer lives on a field record, not as native edge in Kuzu tables.
  • Useful when a field value naturally points to another entity topic.
  • UI graph snapshot synthesizes field_ref edges so refs are visible in visualization.
  • REF_UNCHANGED sentinel allows appending field history without overwriting ref.

Diagram: structural edge vs field ref

The model can use both mechanisms simultaneously for the same topic pair.

Guidance

Use case Prefer Reason
Global graph navigation between entities RELATED Explicit edge semantics and easy neighbor traversal.
Attribute-level pointer inside one field ref_topic_id Keeps reference close to field meaning and revision history.
Need both typed reference and graph traversability Both Use ref_topic_id for schema meaning and RELATED for graph ops.
Relationship reference: structural edges, field refs, and modeling guidance.
