State abstraction and topic model

MemState models memory as a governed evolving state Mt = (Dt, St, Pt). This page specifies what each component is, how topics realize Dt, and why the property graph alone is not sufficient.

State abstraction

Definition 1 — Governed evolving state At time t, an agent's memory state is the tuple Mt = (Dt, St, Pt), where Dt denotes stored informational content organized as semantic units, St denotes the structural organization over that content (relations, temporal order, associations), and Pt denotes policies governing access, ingestion, revision, and forgetting.
Definition 2 — Memory evolution Memory evolves according to a state transition function Mt+1 = U(Mt, It, Rt), where It denotes new inputs or experiences at time t, and Rt denotes internal operations that may alter memory state.
Definition 3 — Evolution policies Pt is a set of typed rules of the form ⟨event, condition, action⟩, where event identifies the operation that triggers evaluation, condition is a predicate over the current memory state Mt, and action is a state-level transition to apply when the condition holds. Policies are declarative: they specify what transitions should occur and when, independently of how operators execute them.
ComponentPurposeMemState instantiation
Dt — content The semantic units that carry the agent's knowledge. Topic nodes. Each topic is one self-contained unit with a title, summary, embedding, and typed fields whose values are maintained as time-ordered histories.
St — structure How units are connected so that changes propagate appropriately. Typed RELATED edges between topics (extension and association) plus field-level ref_topic_id pointers.
Pt — policies Rules governing when and how the state is allowed to change. Declarative thresholds and triggers configured in memstate.config.Policies and evaluated by memstate.reasoner.engine.Reasoner.

Why not a pure property graph, vector DB, or document store

Existing data models instantiate parts of (Dt, St) but leave Pt external, so governed state evolution is not expressible at the data model layer. The table below summarizes the gap each paradigm leaves relative to GEM semantics.

Model Primary limitation under GEM
Property graph (Kuzu, Neo4j) Supports topology as St, but no field-level value history and no first-class Pt. Structure is typed, policy semantics are not.
Vector database Weak St (geometric proximity only). Outdated and updated embeddings coexist; retrieval is similarity-only and read-only.
Document store Document-level overwrite rather than dependency-aware revision. Cross-document structure is denormalized; no retrieval-induced state change.
Temporal database Preserves past states but provides no belief reconciliation: without Pt, earlier values are never marked superseded.

How MemState extends Kuzu

The property graph is the closest classical substrate because it supports typed nodes, typed edges, and indexed properties. MemState builds on Kuzu and extends it along three axes.

  • Self-contained semantic units. Property graphs distribute facts across many nodes and edges. MemState materializes each topic as one node containing its fields, histories, and summary, so one read returns the full context.
  • Field-level value history. Property graphs support temporal edges but not attribute histories. MemState stores a time-ordered history Hi,j = ⟨(v, t, π)⟩ per field, enabling direct history lookup.
  • Co-located semantic ranking. MemState stores embeddings on the topic node itself, so similarity search and graph traversal operate over one consistent record with no cross-store drift.

Topic as semantic unit

Each topic ui ∈ Dt is a self-contained memory unit the agent can read in one access. Using topics instead of entities reduces multi-hop retrieval, lets topic titles guide candidate selection, and allows sparse facts to first coexist in one parent topic before revision promotes a coherent subset into a new topic as knowledge grows.

A topic node carries its fields as versioned JSON, its embedding as a native vector, and its cross-topic links as typed graph edges.

Topic versus entity (modeling boundary)

MemState does not add a separate Entity node type. The topic is the boundary of what is stored as one indexed, versioned unit. Informally, an entity is anything you mean in the world (a person, paper, concept, task); a topic is how that meaning is packaged for the store.

  • Multiple entities may live inside one topic when each is small, needs no standalone identity, and is not heavily shared across the graph.
  • Keep subordinate facts as field values (strings, JSON, lists) while they stay local to the topic's narrative and lifecycle.
  • Split and promote to a new topic when an embedded entity grows complex, is associated with many other topics, or should be queried as a first-class unit. Model the link with RELATED and/or ref_topic_id and move the heavy payload out of the parent topic.

Topic record (schema contract)

Property Type Role in behavior
id STRING (PK) Stable topic identity used by APIs, edges, and field references.
title, summary STRING Human-readable metadata and default text source for embedding generation.
topic_kind STRING Optional classification label for filtering or workflow grouping.
salience, failed_salience DOUBLE Importance and failure-signaling channels used by retrieval reinforcement and forgetting policy.
fields_json STRING (JSON) Typed field map with per-field history Hi,j and optional ref_topic_id.
topic_history_json STRING (JSON) Append-only topic-level event trail (created / salience / meta / embedding / failed_signal).
embedding DOUBLE[] Semantic search vector used by KNN or cosine fallback ranking.
archived BOOLEAN Exclusion flag for default active scans and retrieval candidate generation.
created_at, updated_at STRING (ISO timestamp) Temporal metadata for operations and debugging.

Two distinct history channels

  • Topic history (topic_history_json). Audit events about topic-level changes (creation, salience updates, metadata updates).
  • Field history (fields_json.*.history[]). Value-level temporal revisions for each field; realizes Hi,j = ⟨(v, t, π)⟩ from the paper.

Modeling principles

  • Use scalar topic columns for properties frequently used in filtering, ranking, or maintenance.
  • Use fields for domain-specific attributes that evolve without schema migrations.
  • Use RELATED for structural graph relationships and ref_topic_id for field-level references.
  • Prefer one topic with many small embedded entities until reuse, complexity, or revision boundaries argue for another topic (see Topic versus entity).
