Operation: query (client-accessible)

Query is the read entrypoint for external clients. In code, its objective is to assemble ranked topic bundles from semantic retrieval + optional structural and temporal expansion, then reinforce relevance through salience-touch side effects.

Access and trigger

  • HTTP route: POST /v1/query in memstate.api.main.
  • SDK route: MemoryClient.query().
  • Auth: guarded by API key only when configured.

Objective in code

Objective: produce QueryResponse.candidates[] with controllable stage depth while updating topic salience based on read access to keep active memory discoverable.

Input contract (current implementation)

Field Role Default
qQuery text used for embedding and summary generation.Required
stages[]Execution switches for semantic, structural, temporal.All three
explainEnables full field history emission when temporal stage is present.false
top_kMaximum candidate count considered/returned.8
topic_idsOptional allowlist / fallback source depending on stage setup.None
field_namesOptional field subset for field bundle generation.All topic fields

Stage behavior (current implementation)

Stage Implementation logic Bundle impact
semantic Embed query, rank by vector_knn_topic_ids(); fallback to cosine on loaded vectors. Determines candidate ids and sets similarity.
structural Loads neighbors() and field bundles for each candidate. Populates neighbors[] and fields with current snapshots (or more if temporal+explain).
temporal History expansion occurs only if explain=true. Controls whether field history[] is full or current-only.

Execution sequence

Client -> POST /v1/query -> Executor.query(req)
  -> embed query text
  -> build candidate ids (semantic rank or fallback)
  -> for each candidate topic id
      -> skip missing/archived topics
      -> create TopicBundle metadata
      -> optional neighbors (structural)
      -> optional fields / histories (structural + temporal logic)
      -> salience bump and append topic_history event
  -> build summary_text
  -> return QueryResponse
  -> BackgroundTasks schedules reasoner.run("query_complete")

Important side effects

  • Query is not read-only in implementation: each returned topic gets a salience bump (Policies.query_salience_bump).
  • A topic-history event is appended per touched topic with reason query_bump.
  • These side effects influence future forget decisions in reasoner policy.

Candidate selection details

  • If semantic stage is enabled: ranking starts with vector KNN, then cosine fallback if needed.
  • If semantic stage is disabled and topic_ids provided: candidate list is that allowlist slice.
  • If semantic stage disabled and no allowlist: fallback is first top_k loaded embedding topics.
  • When semantic stage is enabled and topic_ids is provided, allowlist filtering is applied after ranking.

Failure and edge behavior

Condition Behavior
No embeddings availableFalls back to allowlist or empty candidate list depending on request fields.
Archived candidate topicSkipped from result bundles.
Invalid auth keyHTTP 401 before executor is called.
