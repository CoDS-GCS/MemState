Topic model overview

The fundamental memory unit is a Topic. Every topic combines scalar metadata, JSON field records with revision history, and one embedding vector for semantic retrieval.

Topic versus entity (modeling boundary)

Approach. MemState does not add a separate “Entity” node type. Instead, the graph’s topic is the boundary of what is stored as one indexed, versioned unit. Informally, an entity is anything you mean in the world (a person, paper, concept, task); a topic is how that meaning is packaged for the store.

  • A topic behaves like a self-contained record: one node with a fixed scalar envelope and a flexible fields_json map. Multiple entities may be represented inside the same topic when each is small, needs no standalone graph identity, and is not heavily shared across the rest of the graph.
  • Keep subordinate facts as field values (strings, JSON, lists) or compact nested structures while they stay local to that topic’s narrative and lifecycle.
  • Split and promote to a new topic when something that started as an embedded entity becomes complex (many details, long revision history), is associated with many other topics, or should be maintained and queried as a first-class unit. Then model the link with RELATED and/or ref_topic_id on a field, and move the heavy payload out of the parent topic’s inline fields.

How this maps internally. Promotion is a data-modeling / ingest policy choice: tools and reorganize flows create new Topic rows and edges; the store already supports refs and relationships. There is no automatic splitter—agents or batch jobs implement the rule above when writing memory.

Topic record (schema contract)

Property Type Role in behavior
id STRING (PK) Stable topic identity used by APIs, edges, and field references.
title, summary STRING Human-readable metadata and default text source for embedding generation.
topic_kind STRING Optional classification label for filtering or workflow grouping (for example Study sessions).
salience, failed_salience DOUBLE Importance and failure-signaling channels used by query touch logic and maintenance policy.
fields_json STRING (JSON) Typed field map with per-field history and optional ref_topic_id.
topic_history_json STRING (JSON) Append-only topic-level event trail (created/salience/meta/embedding/failed_signal).
embedding DOUBLE[] Semantic search vector used by KNN or cosine fallback ranking.
archived BOOLEAN Exclusion flag for default active scans and query candidate generation.
created_at, updated_at STRING (ISO timestamp) Temporal metadata for operations and debugging.

Conceptual shape

Topic-level structure is fixed for indexing and policy, while field payload remains flexible and versioned.

Two distinct history channels

  • Topic history (topic_history_json): audit events about topic-level changes (creation, salience updates, metadata updates).
  • Field history (fields_json.*.history[]): value-level temporal revisions for each field.

Modeling principles

  • Use scalar topic columns for properties frequently used in filtering, ranking, or maintenance.
  • Use fields for domain-specific attributes that evolve without schema migrations.
  • Use RELATED for structural graph relationships and ref_topic_id for field-level references.
  • Prefer one topic, many small embedded entities until reuse, complexity, or revision boundaries argue for another topic (see Topic versus entity).
