MemState official system documentation

MemState is a single-process memory service built on FastAPI and embedded Kuzu. It stores structured, versioned knowledge as a topic graph and exposes it through three access surfaces on one shared store: /v1 for product memory semantics, /api/ui for low-level graph operations, and /api/llm/chat for tool-based natural language access.

This site is the official engineering reference for maintainers and integrators. Every behavior described here is mapped to the current codebase under src/memstate.

Why a topic graph — not a property graph, vector DB, or document store

MemState does not fit neatly into any of the storage categories that LLM memory systems usually reach for. Understanding why each alternative was rejected explains the shape of the data model.

Property graph (Neo4j / native Kuzu)

A pure property graph places facts on nodes and edges as flat key-value pairs. MemState runs on Kuzu but deliberately does not use Kuzu's native property model for facts. Two things are missing from bare property graphs that matter for memory:

  • No versioning at field granularity. A property graph records the current value of a property. MemState needs to answer "what was the user's job title six months ago and why did it change?" That requires per-field history, provenance, and a why_changed / impact_expected audit trail. These are stored as JSON on the Topic node, not as graph edges.
  • No salience model. Property graphs have no built-in concept of importance or decay. MemState needs to decide which topics to retain under memory pressure, which requires a per-field salience score that rolls up to the topic level and drives automatic archiving.

MemState uses Kuzu for its embedded Cypher engine and native vector type — but wraps it with a semantic layer that the graph engine itself does not provide.

Vector database (Pinecone / Weaviate / Chroma)

Vector databases treat the embedding as the primary data structure. Every record is a vector; search returns nearest neighbors. MemState inverts this: the embedding is metadata on a Topic node, not the organizing principle. Three consequences follow:

  • Graph structure is primary. A topic can be found by traversing RELATED edges from a known topic, not only by semantic similarity. Query combines semantic (vector KNN), structural (neighbor traversal), and temporal (field history) stages in a single response.
  • One embedding per topic, not per chunk. Topics are discrete knowledge units — not document chunks. The embedding is computed once from title + summary and refreshed when those change. There is no chunking pipeline.
  • Retrieval is not the end goal. Query also bumps salience on access, which shapes future forget decisions. The act of reading is part of the memory lifecycle, not just a read operation.

Document store (MongoDB / Firestore)

Document stores version at document granularity — a write replaces the whole document. MemState versions at field granularity: each field carries a history list (newest-first) capped at max_field_history entries (default 500). This means two concurrent facts about the same topic — say, a user's current role and a note about their prior role — are separate versioned fields on one Topic, not two documents.

Document stores also use denormalized references. MemState externalizes cross-topic links as explicit RELATED graph edges with a named kind, so structural queries can traverse the graph without loading every document.

What MemState is instead

A topic is the unit of memory. It has a title, a summary, a single embedding vector, a set of typed versioned fields, a salience score, and outgoing RELATED edges to other topics. The storage model is a property graph node on top of Kuzu, with field versioning and salience logic layered above the graph engine.

A Topic node carries its fields as versioned JSON, its embedding as a native vector, and its cross-topic links as RELATED graph edges.

The semantic layer: ingest, query, revise, forget

The four semantic operations are the only surface a product client needs to know. Each is a deliberate composition of lower-level primitives from GraphStore — the Cypher facade that owns all direct Kuzu access. Understanding the decomposition is key to understanding what MemState actually does and where the decisions live.

Ingest

Ingest is the write path. It has three modes, selected by the caller:

ModeWhen to usePrimitive sequence
new_topic A wholly new piece of knowledge with no existing home embed(title + summary) → create_topic → create_field × N → add_related_edge × M → append_topic_history("created")
extend_topic Adding new facts to an existing topic get_topic → create_field or append_field_history → sync_topic_salience_from_fields → append_topic_history("meta")
version_field Updating one field's value while preserving history get_field_with_history → append_field_history(new entry, prepend) → sync_topic_salience_from_fields

After writing, ingest optionally calls vector_knn_topic_ids to surface similar existing topics — a signal that the caller may be creating a duplicate. This is advisory; deduplication is left to the reasoner's revise pass.

Key decisions in ingest

  • Embedding is computed once from title + summary, not from field values. Fields can be arbitrarily typed and structured; embedding them uniformly is not tractable. The title and summary are the semantic contract of the topic.
  • Field history is prepended, not appended. history[0] is always current. Max history is enforced at write time by slicing to max_field_history (default 500). Old entries fall off silently.
  • Salience is set per field by the caller, not inferred. After any field write, sync_topic_salience_from_fields averages all field saliences to compute the topic salience. The caller is responsible for encoding importance in the field-level salience score it passes.

Query

Query is a multi-stage retrieval. Each stage is independently optional and the results are merged:

StagePrimitiveWhat it returns
semantic vector_knn_topic_ids → get_topic × k Top-k topics by cosine similarity to query embedding (default k=8)
structural list_relationships → get_topic × N 1-hop RELATED neighbors of seed topic(s)
temporal get_field_with_history × F Full field histories with all past values, timestamps, and provenance

After retrieval, query calls bump_field_salience_on_query on every accessed topic, which increments field saliences by query_salience_bump (default 0.1) and re-syncs the topic salience. This makes frequently queried topics more resistant to archiving.

Key decisions in query

  • Vector fallback. If Kuzu's native KNN is unavailable, _load_embedding_vectors reads all non-archived topic embeddings into numpy arrays and _cosine_top_k sorts by dot product after L2 normalization. This is O(N) but avoids a hard dependency on the Kuzu index being available.
  • Reading is a write. The salience bump on query means retrieval is not a pure read. A topic that is regularly queried accumulates salience and is protected from the forget pass. A topic that is never queried decays and eventually gets archived. Access patterns shape retention.
  • Similarity score is cosine, not dot product. Embeddings are L2-normalized before the inner product, so the score is always in [0, 1] regardless of embedding magnitude.

Revise

Revise is an internal maintenance pass triggered automatically after ingest via the reasoner's ingest_complete event. It is never called directly by product clients. Its only implemented policy in v1 is duplicate merging:

  1. Enumerate all topics where two topics share the exact same title (a.title = b.title).
  2. For each duplicate pair, call merge_topics(keep_id, drop_id):
    • Read fields_json from both topics.
    • Merge field maps: fields from drop are added to keep if not already present.
    • Redirect all RELATED edges pointing to drop so they point to keep.
    • Call migrate_legacy_field_chains_to_json to absorb any old-format fields.
    • Delete the drop topic with DETACH DELETE.

Key decisions in revise

  • Duplicate detection is title-exact, not semantic. Two topics with similar but not identical titles will not be merged. This is intentional: semantic deduplication requires a human-in-the-loop or a much higher-confidence signal than embedding proximity.
  • Field merge is additive, not union-with-conflict-resolution. If both topics have a field with the same name, the keep topic's version wins silently. History from the dropped topic's version of that field is not preserved.

Forget

Forget is an internal archiving pass. It runs conditionally: after ingest or query if the topic count is at or above topic_count_soft_limit (default 500), or immediately on a memory_pressure event regardless of count. It does not hard-delete topics.

  1. Find all non-archived topics where salience < forget_salience_threshold (default 0.05).
  2. For each: set archived = true, halve salience as a decay penalty, append a "salience" event to topic history.

Key decisions in forget

  • Archive, not delete. Archived topics remain in the graph and can be retrieved by ID. They are excluded from vector KNN search and from the default query path. This preserves auditability at the cost of storage growth.
  • Salience halving compounds. A topic that survives one forget pass at 0.04 salience becomes 0.02 after archiving. If it is later unarchived (by a direct write), it starts from a lower base — making it more likely to be re-archived on the next pass.
  • The soft limit is a rate control, not a hard cap. The system can hold more than 500 topics. The limit controls when the forget pass is allowed to run, not the absolute count. If every topic has high salience, no archiving occurs regardless of count.

Primitive operations reference

All semantic operations compose from primitives in memstate.store.graph_store. The primitives are the only code that issues Cypher. Understanding them is necessary for debugging, extending, or replacing any semantic operation.

CategoryPrimitiveWhat it does
Topic create_topic INSERT a new Topic node with scalar properties and embedding
get_topic Fetch one topic row including fields_json, topic_history_json, embedding
update_topic_meta SET scalar properties (title, summary, salience, archived, topic_kind); does not touch fields
update_topic_embedding SET embedding vector only; called after title/summary changes
delete_topic DETACH DELETE — removes node and all incident RELATED edges
Field create_field Initialize a named field entry in fields_json with type, optional ref, optional first value
append_field_history Prepend a new FieldHistoryEntry to the field's history list; enforce max_history cap
get_field_with_history Return the full field including all history entries
delete_field Remove field key from fields_json
set_field_ref Point a field's ref_topic_id at another topic (enables expansion)
Structure promote_fields_to_nested_topic Move selected fields to a new child topic; add RELATED edge + optional parent ref field
undo_promote_nested_topic Merge child fields back to parent; delete child topic and RELATED edge
nest_fields_in_topic Collapse selected fields into one json bundle field on the same topic (no new topic)
unnest_fields_in_topic Expand a bundle field back to top-level fields
Edges add_related_edge CREATE/MERGE a RELATED edge with a named kind string
remove_relationship DELETE a specific RELATED edge by (from, to, kind)
list_relationships Return neighbor topic IDs and edge kinds, filtered by direction
Salience sync_topic_salience_from_fields Recompute topic salience as mean of all field saliences; SET on topic node
bump_field_salience_on_query Increment field saliences by bump amount (capped); re-sync topic salience
History append_topic_history Append a TopicHistoryEvent to topic_history_json
merge_topics Merge fields and edges from drop topic into keep topic; DETACH DELETE drop

Source map

  • memstate.api.main: app creation, route registration, auth dependency, health endpoints.
  • memstate.api.deps: singleton wiring for Executor, Reasoner, and shared GraphStore.
  • memstate.core.executor: all /v1 ingest and query semantics, embedding, salience bumping.
  • memstate.store.graph_store and store.kuzu_adapter: all Cypher primitives, schema bootstrap, Kuzu access layer.
  • memstate.reasoner.engine: event-based policy loop (forget + duplicate revise).
  • memstate.llm.chat_api and llm.tool_runner: routed chat and tool execution.
  • memstate.config: all tunable thresholds — forget_salience_threshold, topic_count_soft_limit, max_field_history, query_salience_bump, embedding_dimension.
